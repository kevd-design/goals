Handwritten Journal Entry (10:00 AM):

"To go full simple, or part-way simple?"

I've been weighing the tradeoffs between full simplification and a partial approach:

  • Full simplification would get the project out the door faster
  • This means making the hero static, even though most content is served by the headless CMS
  • For images, it makes sense to still pull from Sanity since their content delivery network would be superior to fully static images
  • I should keep most of the Sanity implementation as-is for images
  • For text that's already configured in the schema, it would be easier to keep using it but make it editable only by myself
  • So my approach is "more simple, but not fully static"
  • The main goal is to disconnect the hero from color mapping and just render the background image as-is
  • I can keep the optimized image and caching functionality, but don't need the decoding or ImageConfigurer components

This hybrid approach will maintain performance benefits while reducing complexity.

Focus Shift:

  • Pivoting from full CMS-driven approach to a simpler MVP with limited Sanity integration
  • Prioritizing completion over advanced features like adaptive colors and accessibility automation
  • Adopting a hybrid approach: static rendering but with Sanity-sourced images for optimization
